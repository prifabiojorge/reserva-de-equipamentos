🏭 PARTE 1 — FILOSOFIA DE ENGENHARIA (Produção + Software)
Antes de escrever uma linha de código, vamos alinhar princípios de engenharia de produção com as decisões técnicas:

Princípio Lean/Produção	Tradução no Software	Aplicação no Projeto
Poka-Yoke (à prova de erros)	Validação no banco + UI que impede conflitos	O sistema bloqueia reserva se já existir outra no mesmo horário/recurso
Kanban (sinal visual)	Dashboard visual tipo quadro semanal	O quadro Kanban se torna uma ferramenta visual de agendamento — como um ToDo bidimensional onde se vê o status de cada tarefa
1
Heijunka (nivelamento de carga)	Visão de ocupação/ociosidade por recurso	Gráficos de calor mostrando % de ocupação de cada recurso por dia/semana
5S (organização)	Estrutura de pastas, código limpo, padrões	Arquitetura modular com separação rigorosa de responsabilidades
Jidoka (parar e corrigir)	Detecção de conflito em tempo real	Notificação instantânea se alguém tenta reservar recurso já alocado
Just-in-Time	Dado certo, hora certa, tela certa	Kanban é um sistema de agendamento para manufatura lean (também chamado just-in-time)
2
 — aqui: o professor vê apenas o que precisa, quando precisa
Kaizen (melhoria contínua)	Versionamento semântico + feedback loop	Sistema de sugestões embutido + analytics de uso
Muda (eliminar desperdício)	Zero WhatsApp, zero papel, zero retrabalho	Substituir 100% do processo manual por digital

Para a maioria das aplicações de negócios — como sistemas de reserva, sites de conteúdo, ou ferramentas simples de produtividade — PWAs conseguem fornecer excelentes experiências offline

generator client {

// ===== USUÁRIOS =====
model Usuario {
  id            String      @id @default(cuid())
  nome          String
  email         String      @unique
  senha         String      // hash bcrypt
  papel         String      @default("PROFESSOR") // PROFESSOR | COORDENADOR | ADMIN
  ativo         Boolean     @default(true)
  criadoEm      DateTime    @default(now())
  atualizadoEm  DateTime    @updatedAt
  reservas      Reserva[]
}

// ===== CATEGORIAS DE RECURSOS =====
model CategoriaRecurso {
  id        String    @id @default(cuid())
  nome      String    @unique  // "Espaço" | "Equipamento" | "Kit"
  icone     String?             // emoji ou nome do ícone
  recursos  Recurso[]
}

// ===== RECURSOS (Equipamentos e Espaços) =====
model Recurso {
  id              String            @id @default(cuid())
  nome            String            // "Laboratório de Informática"
  descricao       String?
  categoriaId     String
  categoria       CategoriaRecurso  @relation(fields: [categoriaId], references: [id])
  quantidadeTotal Int               @default(1)  // Ex: 30 Chromebooks
  quantidadeMinima Int              @default(1)  // Mínimo reservável
  ativo           Boolean           @default(true)
  cor             String?           // Cor no Kanban
  criadoEm        DateTime          @default(now())
  reservas        Reserva[]

  @@index([categoriaId])
}

// ===== CONFIGURAÇÃO DE HORÁRIOS (TURNOS + AULAS) =====
model Turno {
  id        String    @id @default(cuid())
  nome      String    @unique  // "Manhã" | "Tarde" | "Noite"
  ordem     Int                // 1, 2, 3
  horarios  Horario[]
}

model Horario {
  id          String    @id @default(cuid())
  turnoId     String
  turno       Turno     @relation(fields: [turnoId], references: [id])
  numero      Int                // 1º horário, 2º horário...
  horaInicio  String             // "07:00"
  horaFim     String             // "07:50"
  reservas    ReservaHorario[]

  @@unique([turnoId, numero])
  @@index([turnoId])
}

// ===== RESERVAS =====
model Reserva {
  id              String            @id @default(cuid())
  data            DateTime          // Data da reserva (só a data)
  usuarioId       String
  usuario         Usuario           @relation(fields: [usuarioId], references: [id])
  recursoId       String
  recurso         Recurso           @relation(fields: [recursoId], references: [id])
  quantidadeUsada Int               @default(1) // Ex: 8 dos 30 Chromebooks
  observacao      String?
  status          String            @default("CONFIRMADA") // CONFIRMADA | CANCELADA | PENDENTE
  criadoEm        DateTime          @default(now())
  atualizadoEm    DateTime          @updatedAt
  horarios        ReservaHorario[]

  @@index([data, recursoId])
  @@index([usuarioId])
  @@index([data])
}

// ===== JUNÇÃO: RESERVA ↔ HORÁRIOS (N:N) =====
// Permite reservar "todos os horários" ou "2 últimos" etc.
model ReservaHorario {
  id          String    @id @default(cuid())
  reservaId   String
  reserva     Reserva   @relation(fields: [reservaId], references: [id], onDelete: Cascade)
  horarioId   String
  horario     Horario   @relation(fields: [horarioId], references: [id])

  @@unique([reservaId, horarioId])
  @@index([horarioId])
}

// ===== LOG DE AUDITORIA =====
model LogAuditoria {
  id          String    @id @default(cuid())
  usuarioId   String
  acao        String    // "CRIAR" | "EDITAR" | "CANCELAR"
  entidade    String    // "Reserva" | "Recurso"
  entidadeId  String
  detalhes    String?   // JSON com mudanças
  criadoEm    DateTime  @default(now())

  @@index([criadoEm])
}
⚡ PARTE 5 — MOTOR DE DETECÇÃO DE CONFLITOS (Poka-Yoke Digital)
Este é o coração do sistema — a lógica que impede o "cruzamento" de reservas:

TypeScript

// src/actions/conflitos.ts
"use server"

import { prisma } from "@/lib/prisma"

interface CheckConflictParams {
  recursoId: string
  data: Date
  horarioIds: string[]
  quantidadeDesejada: number
  reservaIdExcluir?: string  // Para edição: excluir a própria reserva
}

interface ConflictResult {
  hasConflict: boolean
  conflicts: {
    horarioNumero: number
    horarioLabel: string      // "3º horário (09:30-10:20)"
    professorNome: string
    quantidadeJaReservada: number
    quantidadeDisponivel: number
  }[]
  recursoCapacidadeTotal: number
}

export async function checkConflicts(
  params: CheckConflictParams
): Promise<ConflictResult> {
  
  const { recursoId, data, horarioIds, quantidadeDesejada, reservaIdExcluir } = params

  // 1. Buscar capacidade total do recurso
  const recurso = await prisma.recurso.findUniqueOrThrow({
    where: { id: recursoId }
  })

  // 2. Para cada horário solicitado, verificar reservas existentes
  const reservasExistentes = await prisma.reservaHorario.findMany({
    where: {
      horarioId: { in: horarioIds },
      reserva: {
        data: data,
        recursoId: recursoId,
        status: { not: "CANCELADA" },
        ...(reservaIdExcluir ? { id: { not: reservaIdExcluir } } : {})
      }
    },
    include: {
      horario: { include: { turno: true } },
      reserva: { include: { usuario: true } }
    }
  })

  // 3. Agrupar por horário e somar quantidades
  const conflictMap = new Map<string, {
    horarioNumero: number
    horarioLabel: string
    professores: string[]
    quantidadeTotal: number
  }>()

  for (const rh of reservasExistentes) {
    const key = rh.horarioId
    const existing = conflictMap.get(key) || {
      horarioNumero: rh.horario.numero,
      horarioLabel: `${rh.horario.numero}º horário (${rh.horario.horaInicio}-${rh.horario.horaFim})`,
      professores: [],
      quantidadeTotal: 0
    }
    existing.professores.push(rh.reserva.usuario.nome)
    existing.quantidadeTotal += rh.reserva.quantidadeUsada
    conflictMap.set(key, existing)
  }

  // 4. Detectar conflitos reais (capacidade excedida)
  const conflicts = []
  for (const [, info] of conflictMap) {
    const disponivel = recurso.quantidadeTotal - info.quantidadeTotal
    if (disponivel < quantidadeDesejada) {
      conflicts.push({
        horarioNumero: info.horarioNumero,
        horarioLabel: info.horarioLabel,
        professorNome: info.professores.join(", "),
        quantidadeJaReservada: info.quantidadeTotal,
        quantidadeDisponivel: Math.max(0, disponivel)
      })
    }
  }

  return {
    hasConflict: conflicts.length > 0,
    conflicts,
    recursoCapacidadeTotal: recurso.quantidadeTotal
  }
}

📱 PARTE 11 — PWA (Funcionar no Celular dos Professores)
Uma das funcionalidades destaque das PWAs é sua capacidade de funcionar offline. Isso é alcançado através de service workers — scripts que rodam em segundo plano, separados da página web, habilitando funcionalidades como cache de conteúdo e notificações push. Quando um usuário visita uma PWA pela primeira vez, o service worker faz cache de assets essenciais. Isso significa que mesmo se o usuário perder a conexão de internet, a PWA ainda consegue entregar conteúdo

🎯 RESUMO EXECUTIVO
Este sistema transforma um processo caótico de WhatsApp em um quadro Kanban digital, onde cada "card" é uma reserva, cada "coluna" é um dia da semana, e o motor de Poka-Yoke impede fisicamente que dois professores reservem o mesmo recurso no mesmo horário — assim como numa linha de produção Toyota, onde o erro é impossível por design, não por disciplina humana.

O propósito central do Lean é minimizar atividades de desperdício sem sacrificar produtividade. O objetivo principal é criar mais valor para o cliente sem gerar mais custos.
